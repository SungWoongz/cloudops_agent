# pr

Create a Pull Request for the repository the user is currently working on. The PR body is written in Korean and follows the repository's `PULL_REQUEST_TEMPLATE.md`.

## Usage

```
/pr [base-branch]
```

- `base-branch` (optional): target branch on the upstream repo. Defaults to the upstream's default branch (usually `master` or `main`).

## Preconditions

- The current branch is already committed and pushed. If not, instruct the user to run `/commit` and/or `/push` first.
- `gh` CLI is installed and authenticated.
- When a Jira link is requested, the Atlassian MCP is connected (`mzdevs.atlassian.net`).

---

## 1. Inspect repository state

Run these in parallel:

```bash
git rev-parse --show-toplevel
git branch --show-current
git status --short
git log -1 --pretty=format:'%H%n%s%n%b'
gh repo view --json nameWithOwner,defaultBranchRef,parent
git rev-list --left-right --count @{u}...HEAD 2>/dev/null
```

Validation:

| Check | Action on failure |
|-------|-------------------|
| Inside a git repo | Abort and inform the user |
| HEAD has at least one commit | Abort |
| Current branch tracks a remote and is not ahead of it | Ask the user to run `/push` first |
| `gh auth status` is authenticated | Abort and ask the user to run `gh auth login` |

If the working tree has uncommitted changes, warn the user that those changes will NOT be part of the PR, then ask whether to continue.

## 2. Resolve base repo and branch

1. **Base repo**
   - If the current repo is a fork, use `parent.nameWithOwner` from `gh repo view`.
   - Otherwise, use the current `nameWithOwner`.
2. **Base branch**
   - Use the argument passed to `/pr` if provided.
   - Otherwise, use the upstream's `defaultBranchRef.name`.
3. **Head branch**
   - Current branch. For forks, `gh` will format it as `owner:branch` automatically.

Present the resolved targets to the user before continuing:

```
Base: <owner>/<repo>:<base-branch>
Head: <owner>/<repo>:<head-branch>
```

## 3. Detect PR template

Look for a template in this order (relative to repo root):

1. `.github/PULL_REQUEST_TEMPLATE.md`
2. `.github/pull_request_template.md`
3. `docs/PULL_REQUEST_TEMPLATE.md`
4. `PULL_REQUEST_TEMPLATE.md`

If found, use it as the skeleton **verbatim** — keep all section headers, checkboxes, and HTML comments. If none exists, fall back to the default layout in Section 5.

## 4. Collect commits and diff

```bash
git fetch <base-remote> <base-branch>
git log <base-remote>/<base-branch>..HEAD --pretty=format:'%h %s%n%b%n---'
git diff <base-remote>/<base-branch>...HEAD --stat
git diff <base-remote>/<base-branch>...HEAD --name-only
```

Extract from the output:

- Commit prefixes (`feat`, `fix`, `refactor`, `chore`, `docs`, `test`) → category candidates
- Changed files / line counts
- Bullet points for the `Changes` section
- Notes for reviewers (side effects, known issues)

**Do not infer** anything that is not grounded in the commits, diff, or explicit user input. If something is unclear, ask the user.

## 5. Write the PR body (Korean)

Fill the detected template in Korean while preserving its structure.

### Title rules

- No prefix (`feat:`, `fix:` …). Title convention in the template's top HTML comment takes precedence.
- Korean, within ~50 characters.

### Section rules

| Section | How to fill |
|---------|-------------|
| `Related Issue` | `-` by default. Only set via Section 6. |
| `Category` | Tick `[x]` boxes based on commit prefixes. Ask the user if ambiguous. |
| `Description` | 2–4 Korean sentences on *why* this PR exists, grounded in commits/user input. No speculation. |
| `Changes` | Korean bullet list of concrete changes (files, modules, functions). |
| `Additional Notes` | Reviewer notes, known issues, follow-ups. `-` if none. |

Rules:

- Body text is Korean. File paths, identifiers, commands, and technical terms (API, CI/CD, …) stay as-is.
- Never delete template sections, checkboxes, or HTML comments.
- Empty sections must contain `-` so the section itself is preserved.
- Never include AI attribution (`Claude`, `Generated by Claude Code`, `Co-Authored-By: Claude`, AI …) anywhere in the title or body.

### Default layout (when no template exists)

```markdown
## Related Issue

-

## Description

<2~4 Korean sentences>

## Changes

- <변경 사항 1>
- <변경 사항 2>

## Additional Notes

-
```

## 6. Related Issue (optional)

Fill `Related Issue` via the Atlassian MCP only when at least one of the following holds:

- The user explicitly provides an issue key.
- The user explicitly asks to link the related issue.
- The branch name matches `[A-Z]+-[0-9]+` → confirm with the user before linking.

Procedure:

1. `mcp__atlassian__getAccessibleAtlassianResources` → resolve `cloudId` (default site: `mzdevs.atlassian.net`).
2. `mcp__atlassian__getJiraIssue` → fetch `summary`, `status`, `issuetype`.
3. Render:

   ```markdown
   ## Related Issue

   - [PROJ-123](https://mzdevs.atlassian.net/browse/PROJ-123) — <이슈 요약> (<status>)
   ```

If the lookup fails, tell the user (check the key / permissions) and do not proceed with a broken link.

## 7. Preview and confirm

Show the final draft before creating the PR:

```
Base: <owner>/<repo>:<base-branch>
Head: <owner>/<repo>:<head-branch>

Title: <PR 제목>

Body:
<rendered body>
```

Apply any edits the user requests and re-show the draft. **Never create the PR without explicit confirmation.**

## 8. Create the PR

```bash
gh pr create \
  --repo <base-owner>/<base-repo> \
  --base <base-branch> \
  --head <head-owner>:<head-branch> \
  --title "<PR 제목>" \
  --body "$(cat <<'EOF'
<rendered body>
EOF
)"
```

- Omit `--repo` when the repo is not a fork.
- Use `--head <branch>` (no `owner:`) when base and head are in the same repo.
- Add `--draft` if the user asked for a draft PR.

## 9. Report

```
## PR 생성 완료

| 항목 | 값 |
|------|-----|
| URL | <PR URL> |
| Base | <owner>/<repo>:<base-branch> |
| Head | <owner>/<repo>:<head-branch> |
| Title | <PR 제목> |
| Related Issue | <Jira key 또는 없음> |
| Commits | <n> |
| Files | <n> (+<added> / -<removed>) |
```

---

## Important Guidelines

- **Never invent content.** Only use facts grounded in commits, diff, template, or explicit user input. Ask when unclear.
- **Preserve the template.** Keep every section header, checkbox, and HTML comment exactly as in `PULL_REQUEST_TEMPLATE.md`.
- **Korean body, no-prefix Korean title.** Technical identifiers stay in their original form.
- **Require user confirmation** (Section 7) before running `gh pr create`.
- **Do not auto-guess the Jira issue.** Only link when the user asks or confirms a branch-name match.
- **Atlassian default site:** `mzdevs.atlassian.net`.
- **Handle fork repos:** base = parent (`parent.nameWithOwner`), head = `owner:branch` on the fork.
- **No AI attribution.** Never add `Claude`, `Generated by Claude Code`, `Co-Authored-By: Claude`, or similar strings to the title or body.
